# Ручные тесты Restobook (для отчёта и скриншотов)

Документ для **ручного** прогона в браузере. Учётные данные и роли — в [`docs/DEMO_USERS.md`](docs/DEMO_USERS.md). Общий пароль сида: **`Password1`**.

**Предусловия для всех сценариев:** запущены API (монолит, например `go run ./cmd/api` в `backend`) и фронт (`npm run dev` в `frontend`), в браузере открыт адрес фронта (часто `http://localhost:5173`).

В отчёте для каждого теста удобно указать: **идентификатор**, **краткое название**, **результат (пройден / не пройден)**, **номер скриншота** и одно-два предложения вывода по сути проверки.

---

## MT-01. Гость: бронирование столика через визард

**Цель проверки:** убедиться, что гость может пройти типовой сценарий: выбор ресторана/даты/времени → схема зала → выбор стола → данные брони → успешное создание брони (или шаг до оплаты депозита, если в вашей конфигурации требуется оплата).

**Роль:** гость (`client`).

| Поле | Значение |
|------|----------|
| Логин | `client@demo.ru` |
| Пароль | `Password1` |

**Шаги**

1. Выйти из других сессий при необходимости. На странице входа авторизоваться под гостем.
2. Перейти к бронированию: с главной выбрать ресторан или открыть страницу ресторана, например **`/restaurant/trattoria-tverskaya`** (slug из сида).
3. Выбрать дату **не в прошлом**, интервал времени из предложенных.
4. На шаге со схемой зала выбрать **свободный** стол (не заблокированный).
5. Заполнить комментарий при необходимости, подтвердить бронь; при запросе депозита — дойти до экрана оплаты или успешного подтверждения (зафиксировать фактическое поведение).

**Ожидаемый результат:** ошибок валидации нет; бронь создана или пользователь корректно доведён до шага оплаты/подтверждения; в разделе **«Мои брони»** появляется новая запись (если бронь уже подтверждена в UI).

**Скриншоты для отчёта**

1. Экран выбора даты/времени или ресторана.  
2. Схема зала с выбранным столом.  
3. Итог: подтверждение брони или список «Мои брони» с новой бронью.

**Формулировка в отчёте (пример):**  
*«Тест MT-01 подтверждает работоспособность гостевого сценария бронирования: навигация по визарду, интерактивная схема зала и сохранение брони в учётной записи гостя.»*

## MT-02. Администратор зала: панель броней и чек-ин

**Цель проверки:** убедиться, что администратор видит список броней по ресторану и может выполнить операцию **чек-ин** (или аналог смены статуса), характерную для эксплуатации зала.

**Роль:** администратор (`admin`).

| Поле | Значение |
|------|----------|
| Логин | `admin@demo.ru` |
| Пароль | `Password1` |

**Шаги**

1. Выйти из сессии гостя. Войти под `admin@demo.ru`.
2. Перейти в раздел **«Админ»** (`/admin`) — по умолчанию откроется список броней ресторана администратора.
3. Найти бронь в статусе, допускающем чек-ин (например подтверждённая на сегодня/ближайшее время — при отсутствии подходящей записи зафиксировать в отчёте «данных не хватило» и приложить скрин списка).
4. Выполнить **чек-ин** для выбранной брони.

**Ожидаемый результат:** список броней загружается без ошибки; после чек-ина статус брони обновляется в интерфейсе (или отображается сообщение об успехе).

**Скриншоты для отчёта**

1. Панель броней с видимым фильтром/списком.  
2. До и после чек-ин (или одно окно с явным изменением статуса).

**Формулировка в отчёте (пример):**  
*«Тест MT-02 демонстрирует готовность админ-интерфейса к операционной работе: просмотр броней и смена статуса (чек-ин) без сбоев клиентской части.»*

## MT-03. Владелец: кабинет и аналитика (или настройки)

**Цель проверки:** убедиться, что владелец ресторана получает доступ к кабинету и к аналитическому/настроечному разделу, недоступному гостю.

**Роль:** владелец (`owner`).

| Поле | Значение |
|------|----------|
| Логин | `owner@demo.ru` |
| Пароль | `Password1` |

**Шаги**

1. Выйти из админа. Войти под `owner@demo.ru`.
2. Перейти в раздел **«Владелец»** (`/owner`): дашборд, аналитика, настройки — как в вашем UI.
3. Кратко открыть **настройки** ресторана или раздел персонала (без обязательного сохранения), убедиться, что страницы открываются.

**Ожидаемый результат:** разделы владельца доступны; нет «403» / пустого экрана из-за ошибки прав.

**Скриншоты для отчёта**

1. Главный экран кабинета владельца или график аналитики.  
2. Любой второй экран (настройки или персонал).

**Формулировка в отчёте (пример):**  
*«Тест MT-03 подтверждает разграничение ролей: владелец получает доступ к управленческим экранам (аналитика, настройки), что соответствует модели доступа Restobook.»*

## MT-04. Контроль доступа: гость не видит админ-панель

**Цель проверки:** убедиться, что при прямом переходе по URL гостю **недоступны** экраны администратора/владельца (редирект, 403 или «нет доступа»).

**Роль:** гость.

| Поле | Значение |
|------|----------|
| Логин | `client1@demo.ru` (или `client@demo.ru`) |
| Пароль | `Password1` |

**Шаги**

1. Войти под гостем.
2. Вручную в адресной строке открыть, например, **`/admin`** или **`/owner`** (полный URL: `http://localhost:5173/admin` при локальном Vite).
3. Зафиксировать поведение: по текущей логике фронта гость с сессией **редиректится на главную** `/` (роль не входит в `RoleGate`), а не получает панель админа или владельца.

**Ожидаемый результат:** гость **не** получает полноценный функционал админа/владельца.

**Скриншоты для отчёта**

1. Адресная строка с «чувствительным» URL и видимый результат (сообщение или редирект на безопасную страницу).

**Формулировка в отчёте (пример):**  
*«Тест MT-04 проверяет защиту маршрутов: учётная запись с ролью гостя не получает доступ к привилегированным разделам при прямом вводе URL.»*

## MT-05. Официант: «Мои столы» и контекст смены

**Цель проверки:** убедиться, что официант видит рабочий экран со столами/сменами, соответствующий роли `waiter`.

**Роль:** официант.

| Поле | Значение |
|------|----------|
| Логин | `waiter@demo.ru` |
| Пароль | `Password1` |

**Шаги**

1. Выйти из гостевой сессии. Войти под `waiter@demo.ru`.
2. Перейти по пункту **«Официант»** в шапке или открыть **`/waiter`**.
3. При наличии активных броней на столах — открыть карточку брони или стол (без обязательного изменения данных).

**Ожидаемый результат:** интерфейс официанта загружается; отображаются столы или осмысленное пустое состояние («нет активных»), без необработанной ошибки.

**Скриншоты для отчёта**

1. Список столов / смен.  
2. При возможности — детальная карточка стола или брони.

**Формулировка в отчёте (пример):**  
*«Тест MT-05 подтверждает работоспособность официантского рабочего места: отображение столов и привязка к ресторану из сида.»*
