## Брони и платежи (seedLifeData)

Брони **не привязаны к фиксированному email в коде по одному правилу «этот клиент = эта бронь»**. Алгоритм:

1. Собирается **упорядоченный по email** список всех пользователей с ролью `client`.
2. Столы берутся из **общей** таблицы `tables` (порядок: `hall_id`, `table_number`) — то есть брони распределены по **всем залам и ресторанам**.
3. Для каждой генерируемой брони задаются индексы «стол» и «клиент»; клиент выбирается как `clients[ci % len(clients)]`.

Поэтому у конкретного `clientN@demo.ru` набор ресторанов, статусов и дат **зависит от числа клиентов и порядка сортировки email** и может меняться, если список клиентов изменён.

### Что гарантированно появляется в данных (типы сценариев)

- Много **завершённых** визитов (`completed`) с депозитом (`payments`, статус `succeeded`), часть с **частичным возвратом** (`refund_amount_kopecks`).
- Несколько броней со статусами **отмена** / **no-show** (без успешного депозита в тех же правилах, что в сиде).
- **«Сегодня»** (календарный день по `Europe/Moscow`): три брони с разными статусами — например **обед `seated`**, **вечер `in_service`**, ещё одна **`confirmed`** (конкретные столы и клиенты — по индексам в коде).
- **Будущие** брони (`confirmed`), часть в **`pending_payment`** с платежом `pending`.
- Одна бронь, созданная от имени **админа** (`created_by: admin`) — «Корпоратив (админ)».

### Официанты и заметки

- К броням подставляются `waiter@demo.ru` и `waiter2@demo.ru` по очереди (в зависимости от сценария).
- К части броней добавляются **заметки официанта** (`waiter_notes`) из фиксированного набора фраз.
- Для активных по времени визитов столы могут помечаться как **occupied**.

## Заказы по меню и счёт (tab) после визита

Функция `seedClosedOrdersForCompleted` для части завершённых броней **без** существующего заказа:

- создаётся **`reservation_orders`** со статусом `closed`;
- в **`order_lines`** попадают **2 случайные позиции** из меню **того ресторана**, где был стол (выбор `ORDER BY random()`);
- добавляется **платеж** с `purpose: tab` на сумму строк.

То есть у разных завершённых визитов в разных ресторанах будут **разные блюда** из меню этого заведения, а не фиксированный набор на человека.

## Как пользоваться этим файлом

- Для проверки **владельца** входите под `owner@demo.ru` / `owner2@demo.ru` / `owner3@demo.ru` и смотрите свой ресторан из таблицы выше.  
- Для **гостя** — `client@demo.ru` или любой `clientN@demo.ru`; в «Мои брони» отобразится смесь прошлых, сегодняшних и будущих броней согласно алгоритму.  
- Для **официанта** — `waiter@demo.ru` (Траттория) или `waiter3` / `waiter4` для других заведений.  
- Для **админа зала** — `admin@demo.ru` (Траттория).

При необходимости точного соответствия «email → id брони» смотрите данные в PostgreSQL после запуска сида.
