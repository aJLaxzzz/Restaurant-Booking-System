## Сид: брони, заказы, уведомления

- Если в БД уже есть строки в `reservations`, блок «живых» демо-броней и заказов **может не пересоздаваться**. Полный нулевой набор — пустая БД или сброс тома PostgreSQL.
- **Распределение броней по клиентам** в `seedLifeData`: клиенты сортируются по email, столы — по залам; конкретный `clientN@demo.ru` ↔ бронь зависит от числа клиентов и порядка сортировки.
- В данных обычно есть завершённые визиты, отмены, no-show, брони «сегодня» (день по Europe/Moscow), будущие `confirmed` / `pending_payment`, бронь «Корпоратив (админ)».
- Для части завершённых визитов создаются заказы (`reservation_orders` + `order_lines`) и платёж счёта (`purpose: tab`).
- Для интеграционных сценариев удобны **`client2@demo.ru`**, **`client3@demo.ru`** (см. `backend/test/integration/api_integration_test.go`).
- Записи в **`notifications`** — демо напоминаний (реальные SMS в проекте не отправляются, см. notify-worker).

## Сценарии проверки UI / API

### Администратор системы

1. Войти как **`superadmin@demo.ru`** / `Password1`.
2. Открыть **`/superadmin`**: заявки владельцев, пользователи, рестораны, брони (нужен выбор ресторана и `restaurant_id` в запросах к общим API), меню, счёт по UUID брони, глобальные настройки.

### Оплата счёта после визита

1. **`client@demo.ru`** — бронь, депозит (или нулевой депозит в настройках), подтверждение.
2. **`admin@demo.ru`** — чек-ин, посадка (`seated`).
3. **`waiter@demo.ru`** — позиции в заказе, отметка «принесено», при необходимости checkout счёта.
4. Для демо-оплаты: **`POST /api/payments/checkout/{id}/simulate`** (от пользователя, которому принадлежит платёж).

### Заказ гостя («Мои брони»)

**`client@demo.ru`**: активные и прошлые брони; оплата счёта, если открыт счёт и статус `seated` / `in_service`.

### Официант

**`waiter@demo.ru`** (или официант нужного ресторана): страница официанта, добавление блюд, mark-served. После заказа гостя за столом ему автоматически назначается официант из графика смены (приоритет у `@demo.ru`).

### Владелец

**`owner@demo.ru`**: контакты, часы, настройки бронирования; проверка **`GET /api/booking-defaults`** и страницы зала **`/hall`**.

### Редактор схемы зала (владелец / админ)

На **`/hall`** в режиме редактирования:

- **Стены** — сегменты и кривые; **интерьер** — столы, двери, окна, зоны.
- **«Зона по клику»** строит контур по **стенам и краю чертежа** (сеточный flood fill) или по уже сохранённым полигонам зон / контурам помещений; при **двух зонах друг в друге** клик внутри **внутренней** выбирает меньшую зону, чтобы разметить их как **две отдельные** зоны с разными подписями.
- У **вложенного прямоугольника**, где часть стен совпадает с периметром большого зала, контур зоны опирается на **стыки сегментов**; при странном заливе проверьте, что внутренние стены **сходятся в углы** (сетка 20 px) и нет щелей.
- **Откат:** ⌘Z / Ctrl+Z (не в полях ввода) — в том числе отмена последнего **создания стола** (стол удаляется и на сервере).
- **Город** ресторана в настройках владельца попадает в фильтр **городов на главной** после сохранения и обновления списка (вкладка / возврат на главную).

### Каталог без входа

Главная и карточка ресторана: телефон, часы, адрес, email из данных заведения.

## Монолит и шлюз (локальная разработка)

- По умолчанию: **`scripts/dev-restart.sh`** — монолит на порту 8080.
- Микросервисы за nginx: **`scripts/dev-restart.sh --gateway`** (тот же 8080, прокси Vite не меняется).

При работе через шлюз часть путей проксируется на разные сервисы; панель администратора системы и загрузки завязаны на reservation-сервис (см. `nginx/nginx.conf`).
